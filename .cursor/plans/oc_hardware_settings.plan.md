# Задача к реализации: конфигурация оборудования OpenConnect

## Решения пользователя (зафиксировано)

- **Объём:** всю описанную ниже реализацию выполнять **одним PR**.
- **Хранилище:** основной конфиг — **`StatefulService` + `FSPersistence`** (LittleFS JSON).
- **Единый UI без дубля переключателей** для включения «железа» (см. ниже порядок вкладок).

---

## Структура вкладок Open Connect (уточнено)

1. **Первая вкладка** — **«Оборудование»** (или эквивалентное название): только **включение/отключение** блоков железа, адреса I2C, флаги PCF8574 и т.д. **Не** смешивать с порогами датчиков.
2. **Отдельная вкладка** — **«Датчики давления»** (или переименование текущей): только **настройки работы** этого железа (пороги, зоны — как сейчас через [`SensorsSettings`](interface/src/lib/components/OCSettings/SensorsSettings.svelte) для `pressure`).
3. **Условное отображение:** если соответствующее железо **выключено** в конфиге оборудования — **не показывать** связанный раздел настроек этого железа (вкладку целиком исключить из списка **или** не рендерить содержимое настроек; недопустимо показывать форму порогов для отключённого блока).

Порядок в [`Settings.svelte`](interface/src/routes/oc/settings/Settings.svelte): вкладка **«Оборудование»** идёт **первой** в массиве табов (после статичных вроде «Профили» — уточнить желаемый порядок: обычно «Оборудование» сразу после профилей или самой первой пользовательской вкладкой).

## Перезагрузка: диалог и отказ «позже» (уточнено)

### Паттерн из проекта

Использовать тот же подход, что в **[`interface/src/routes/system/status/SystemStatus.svelte`](interface/src/routes/system/status/SystemStatus.svelte)**:

- **`modals.open(ConfirmDialog, { … })`** из `svelte-modals`;
- **`POST /rest/restart`** с теми же заголовками авторизации, что и в `postRestart()` в этом файле;
- Тексты кнопок через `labels.cancel` / `labels.confirm` по образцу `confirmRestart()`.

После успешного **`PUT`** конфига железа, если ответ содержит **`needsReboot: true`**, показать диалог подтверждения перезагрузки по аналогии (заголовок/текст адаптировать под «изменения вступят после перезагрузки»).

### Обязательный баннер при отказе от немедленной перезагрузке

- Пользователь должен иметь возможность выбрать **«Перезагрузить позже»** / закрыть диалог **без** перезапуска.
- При этом на экране настроек Open Connect (как минимум) должна оставаться **обязательная заметная плашка** (например DaisyUI **`alert-warning`**): фиксируется, что **были сохранены важные изменения, требующие перезагрузки контроллера**, пока перезагрузка не выполнена.

**Рекомендуемая реализация состояния «ожидается перезагрузка»:**

- либо флаг в ответе **`GET /rest/oc/hardware-config`**, например **`pendingReboot: true`**, который прошивка выставляет после успешного сохранения «жёстких» полей и сбрасывает после фактической перезагрузки (volatile не нужен — после reboot состояние пропадает само);
- либо комбинация с локальным флагом во фронте до первой успешной перезагрузки — но для устройств без последующего доступа надёжнее **источник на контроллере**, чтобы баннер был правдивым после повторного захода в UI.

Текст плашки — нейтральный, без «§», по правилам проекта.

## Принцип «без дублирования выключателей»

Источник правды для включения блоков — **JSON конфиг железа** + синхронизация с **`SubsystemManager`** без второго независимого тумблера на вкладке «Датчики давления».

Тумблер включения давления/BMP581 размещается **только** во вкладке «Оборудование». Вкладка «Датчики давления» — только параметры работы при **включённом** железе.

---

## Ограничение SubsystemManager и перезагрузка

Условная **`registerSubsystem<AtmosphericSubsystem>()`** до `SubsystemManager::begin()` по конфигу. Изменения «жёстких» полей → **`needsReboot`**; без перезагрузки не гарантировать применение.

---

## Миграция (уточнено — **обязательна**, критичный риск)

Разовое применение на устройствах **без дальнейшего физического доступа** — миграция должна быть **обязательной**, **идемпотентной** и **проверяемой**.

### Требования

1. **Триггер:** при первом запуске прошивки с новой логикой — если файл конфига железа отсутствует или **`schemaVersion`** ниже текущего, выполнить миграцию **до** чтения конфига кодом `SsvcOpenConnect`/`SubsystemManager` (порядок старта зафиксировать в коде).
2. **Источник legacy-данных:** текущее NVS-хранилище **`SubsystemManager`** (`subsystem`/`config` через [`GlobalConfig`](lib/ssvcOpenConnect/core/GlobalConfig/GlobalConfig.h)) — перенести соответствующие поля (в т.ч. **`atm_sensor`**) в JSON конфига без потери смысла.
3. **Целостность:**
   - запись JSON **атомарно** (временный файл + rename / запись целиком в один проход по принятому в проекте стилю);
   - после успешной записи — **пометить миграцию выполненной** (поле в JSON **`schemaVersion`** и/или отдельный ключ **`migrationCompleted`** / bump версии схемы), чтобы повторный запуск не перетирал пользовательские правки;
   - при ошибке записи — **не** удалять старые данные NVS до успеха; логировать уровнем **ERROR** в UART с понятным префиксом (`OC_HW_MIGRATION`) для диагностики на расстоянии.
4. **Согласованность NVS после миграции:** явное правило в реализации: либо однократная синхронизация NVS из нового JSON, либо прекращение использования старого ключа для полей, перенесённых в JSON (избежать двух источников после миграции).
5. **Тестовые сценарии (обязательно описать в PR / чеклисте):**
   - устройство только с legacy NVS, без файла конфига;
   - устройство с уже частично созданным файлом;
   - повторная загрузка после успешной миграции (идемпотентность);
   - отключённое давление в legacy → корректное отражение в JSON и отсутствие регистрации BMP581 после reboot.

## Тестирование / приёмка

- Миграция legacy → JSON на «холодном» устройстве без FS-файла.
- Повторный boot не ломает конфиг.
- Баннер «нужна перезагрузка» при **Перезагрузить позже** и его исчезновение после `POST /rest/restart`.
- Вкладка настроек давления **не видна** (или пустое состояние без полей порогов) при выключенном железе.
- Сборка **`kincony-kc868-a6`**, **`PINOUT_USE_GPIO`**.

## Список задач по порядку выполнения

Задачи выстроены так, чтобы миграция и модель данных были определены до любого потребления конфига при старте.

1. **Спецификация данных** — зафиксировать структуру `OpenConnectHardwareConfig` (C++): поля из плана (`schemaVersion`, флаги давления/реле, адреса I2C, `pendingReboot`), значения по умолчанию, имена файла LittleFS и ключа REST (`/rest/oc/hardware-config`).
2. **Зеркало типов во фронте** — интерфейсы TypeScript и константы схемы в [`interface/src/lib/types/ssvc.ts`](interface/src/lib/types/ssvc.ts) (или отдельный файл типов OC).
3. **Миграция legacy → JSON** — модуль однократной миграции: чтение NVS `subsystem`/`config` через [`GlobalConfig`](lib/ssvcOpenConnect/core/GlobalConfig/GlobalConfig.h), сборка начального JSON, атомарная запись файла, идемпотентность по `schemaVersion`/флагу завершения, при ошибке — не ломать NVS, лог **`OC_HW_MIGRATION`** уровнем ERROR.
4. **Порядок загрузки** — вызвать миграцию **до** первого использования конфига железа кодом [`SsvcOpenConnect`](lib/ssvcOpenConnect/core/SsvcOpenConnect.cpp) / [`SubsystemManager`](lib/ssvcOpenConnect/core/SubsystemManager/SubsystemManager.cpp) (явный порядок в `begin`).
5. **Сервис конфигурации** — класс на базе **`StatefulService<OpenConnectHardwareConfig>`** + **`FSPersistence`**: `readFromFS`/`update`, сериализаторы JSON, установка и сброс **`pendingReboot`** при сохранении жёстких полей и после reboot.
6. **Условная регистрация подсистем** — в **`SsvcOpenConnect::begin`**: читать эффективный конфиг; регистрировать **`AtmosphericSubsystem`** только при включённом давлении; согласовать начальное состояние **`atm_sensor`** с JSON и правило после миграции NVS (единый источник правды).
7. **Реле PCF8574 и GPIO** — подставить адрес/флаги из сервиса вместо голых макросов в **[`RelayPortCoordinator`](lib/ssvcOpenConnect/core/RelayPortCoordinator/RelayPortCoordinator.cpp)**, **[`RelayRuleEngine`](lib/ssvcOpenConnect/core/RelayRuleEngine/RelayRuleEngine.cpp)** (metadata и логика), **[`OpenConnectHandler`](lib/ssvcOpenConnect/API/handlers/OpenConnectHandler/OpenConnectHandler.cpp)** relay API; при **`PINOUT_USE_GPIO`** отключить/скрыть опции PCF через **`capabilities`**; при **`userRelayPcfEnabled`** false не поднимать нужные компоненты согласно текущей точке входа инициализации.
8. **REST API** — `GET`/`PUT` аппаратного конфига: валидация диапазонов I2C, ответы **`capabilities`**, **`needsReboot`**, после `PUT` жёстких полей — **`pendingReboot`**; регистрация в **[`HandlerRegistrator.cpp`](lib/ssvcOpenConnect/API/HandlerRegistrator.cpp)** и делегирование в обработчик OpenConnect (по принятому стилю проекта).
9. **Клиент API** — функции [`apiFetch`](interface/src/lib/api/ssvcApi.ts) для конфига железа и при необходимости обёртка над **`POST /rest/restart`** для диалога перезагрузки.
10. **Компонент «Оборудование»** — например **`HardwareConfigTab.svelte`**: форма вкладки «Оборудование», карточки полей, disabled-состояния из **`capabilities`**.
11. **Интеграция вкладок** — **[`interface/src/routes/oc/settings/Settings.svelte`](interface/src/routes/oc/settings/Settings.svelte)**: первая пользовательская вкладка «Оборудование» после «Профили» (или согласованный порядок); фильтрация вкладки «Датчики давления»/`atm_sensor`, если давление выключено в конфиге; сохранить мобильный паттерн как у SSVC (**`tabs-lift`**, **`SettingsMobileScroll`**, `?tab=`).
12. **Перезагрузка и баннер** — после `PUT` при **`needsReboot`**: диалог по образцу **[`SystemStatus.svelte`](interface/src/routes/system/status/SystemStatus.svelte)** (`ConfirmDialog`, **`POST /rest/restart`**); действие «перезагрузить позже» + **`alert-warning`** по **`pendingReboot`** из **`GET`**.
13. **Убрать дубли управления** — вкладка/компонент настроек давления: только пороги и параметры работы без тумблера включения железа; проверить **[`SensorsSettings`](interface/src/lib/components/OCSettings/SensorsSettings.svelte)** и **`toggleSubsystem`** для **`atm_sensor`**, чтобы не конфликтовать с вкладкой «Оборудование».
14. **Приёмочные проверки** — пройти чеклист раздела «Тестирование / приёмка» плана и сценарии миграции из раздела «Миграция»; сборка **`kincony-kc868-a6`** и сценарий **`PINOUT_USE_GPIO`**.
