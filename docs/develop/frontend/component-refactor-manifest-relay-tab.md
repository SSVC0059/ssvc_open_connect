# Component Refactor Manifest: Relay Rule Editor

## Назначение
Этот документ фиксирует эталонную схему рефакторинга UI-компонентов по образцу `ProfileOpenConnectRelayTab.svelte`.

Цель:
- дать ИИ и разработчику единую начальную точку;
- сохранить UX-логику при полном пересмотре разметки и стилей;
- стандартизировать визуальную плотность, иерархию и поведение на разных экранах.

---

## 1) Контекст компоненты

Компонента: `interface/src/lib/components/profiles/ProfileOpenConnectRelayTab.svelte`

Сценарий пользователя:
1. Просмотреть список правил (сканирование).
2. Выбрать правило (навигация).
3. Отредактировать правило по шагам `Состояние -> Если -> Тогда`.
4. Выполнить вторичное действие (копия, удаление, изменение порядка).

Базовый принцип:
- список = навигатор;
- редактор = место редактирования;
- технические детали = низкий приоритет (свернуто).

---

## 2) Свод правил UI/UX (обязательные)

### 2.1 Информационная архитектура
- Один primary action в шапке (`+ Добавить ...`).
- Сводка и служебные подсказки должны быть компактными, без конкуренции с рабочей зоной.
- Вторичные действия запрещено распылять по карточкам списка, они должны быть рядом с активным объектом (в редакторе).

### 2.2 Список элементов
- Список отображает только обзор: номер, тип/статус, короткое условие, короткое действие.
- Высота элемента списка должна быть стабильной (ellipsis для длинных строк).
- В списке не размещать мелкие action-группы из 3+ кнопок.

### 2.3 Редактор
- Обязательная шаговая структура (минимум 3 шага, если применимо): `Состояние`, `Условие`, `Действие`.
- Поля внутри шага группируются логически; без лишних рамок вокруг каждого мини-блока.
- Подтверждение удаления должно явно содержать контекст (`Удалить правило #N?`).

### 2.4 Управление приоритетом/порядком
- Не показывать reorder-контролы постоянно.
- Использовать отдельный режим (`Изменить порядок`) или drag-mode.
- Перемещение доступно только для выбранного элемента.

### 2.5 Адаптив
- На узких экранах запрещен параллельный двухколоночный скролл.
- Нужен явный mobile flow: `list -> editor`.
- В мобильном редакторе должна быть компактная sticky-навигация (`К списку`, `Назад`, `Далее`).

### 2.6 Контраст и доступность
- Не использовать массово low-opacity 0.5/0.6 для рабочего текста.
- Минимальный размер интерактивных контролов должен быть touch-дружелюбным.
- Нельзя делать контейнер `role="button"` с вложенными активными элементами без четкой focus/keyboard модели.

## 3) Визуальный манифест (look-and-feel contract)

### 3.1 Визуальная иерархия
- Уровень 1: заголовок + primary action.
- Уровень 2: список и редактор.
- Уровень 3: технические детали и служебные пояснения.

### 3.2 Пространство и ритм
- Использовать шаг отступов: `8 / 12 / 16` px (или эквивалент rem).
- Сначала уменьшать количество контейнеров, затем настраивать отступы.
- Избегать каскада из нескольких вложенных "box-in-box" без функциональной причины.

### 3.3 Типографика
- Основной рабочий текст: читаемый, без сильного приглушения.
- Подписи полей: на полшага меньше, но с достаточным контрастом.
- Статусы и микротексты не должны быть важнее основного содержания.

### 3.4 Плотность
- Компактность достигается сокращением "лишних" элементов, а не уменьшением шрифтов до микроскопических.
- Если область перегружена, сначала переносить редкие действия в контекстную зону.

---

## 4) Технический манифест для логики данных

### 4.1 Single source of truth
- Для селектов и привязок использовать внутренний ключ доменной модели (например, `targetBit`), а не визуальный индекс (`index`) при их расхождении.
- Любое отображаемое значение должно восстанавливаться из одного стабильного поля.

### 4.2 Стабильность выбора
- `value` в `select` должен совпадать с типом и ключом `option value`.
- Нельзя смешивать "идентификатор отображения" и "идентификатор состояния".

### 4.3 Безопасные fallback
- Сначала точное сопоставление.
- Затем fallback-нормализация (например, clamp), если доменная модель допускает частично устаревшие значения.

## 5) Анти-паттерны (запрещено повторять)

- Список-редактор-гибрид, где карточка одновременно и навигация, и full control panel.
- Дублирование одного и того же переключателя в разных местах без явной причины.
- Постоянные "Вверх/Вниз/Копия/Удалить" в каждой строке длинного списка.
- Технические JSON-блоки в основной рабочей зоне по умолчанию.
- Адаптив "в одну колонку" только на CSS без пересмотра сценария взаимодействия.

---

## 6) Definition of Done для AI-рефакторинга

Результат считается готовым, если:
- пользователь на desktop выполняет базовый сценарий без избыточного скролла;
- список читается как обзор, а не как пульт управления;
- вторичные действия локализованы рядом с выбранной сущностью;
- mobile flow последовательный (`список -> редактор`);
- не сломана доменная логика выбора/привязки значений;
- в измененных файлах нет новых linter errors.

## 7) Шаблон задачи для ИИ (копировать как старт)

```md
Задача: выполнить full UX/UI рефакторинг компоненты <path>.

Ограничения:
1. Сохранить бизнес-логику и типы данных.
2. Перестроить интерфейс по схеме:
   - список как навигатор;
   - редактор как пошаговая зона редактирования;
   - вторичные действия только в редакторе выбранного элемента.
3. Устранить неэффективное использование пространства:
   - уменьшить визуальный шум;
   - стабилизировать высоту элементов списка;
   - увести тех. детали в collapsible низкого приоритета.
4. Реализовать mobile flow: list -> editor с быстрым возвратом.
5. Проверить привязки селектов и ключей состояния (single source of truth).
6. После правок запустить lint для измененных файлов и исправить новые ошибки.

Ожидаемый результат:
- краткое описание архитектурных изменений;
- список UX-проблем "до -> после";
- перечень измененных файлов;
- подтверждение линтера.
```

## 8) Что переиспользовать в других компонентах

Подходит для:
- редакторов правил/сценариев/автоматизаций;
- настроек с "список объектов + детальная правая панель";
- сложных форм с несколькими типами условий.

Неподходящий случай:
- примитивные формы из 1-3 полей без навигационного списка.
